# Манилов — система бронирования столов

Внутреннее приложение для персонала кафе. Официанты и админы бронируют
столы, видят занятость в реальном времени, отмечают предзаказы.

## Стек
React + Vite, Supabase (Auth + PostgreSQL + Realtime), деплой на Vercel.

## Роли
- **admin** (2 человека) — всё, включая удаление броней и правку схемы зала
- **staff** (4 человека) — создание и редактирование броней

## Запуск локально
1. `npm install`
2. Скопируй `.env.local.example` в `.env.local`, впиши URL и публичный ключ
   из Supabase → Project Settings → API.
3. В Supabase → SQL Editor выполни `supabase/migration.sql`.
4. Создай сотрудников: Authentication → Add user (email + пароль).
5. Назначь роли и имена (SQL Editor):
   ```sql
   update public.profiles set role = 'admin', full_name = 'Иван' where id = 'uuid';
   update public.profiles set full_name = 'Мария' where id = 'uuid-staff';
   ```
6. Отключи открытую регистрацию: Authentication → Providers → Email →
   выключи "Enable signup". Тогда заходить смогут только созданные тобой аккаунты.
7. `npm run dev`

> Если переменные окружения не заданы, приложение не падает с белым экраном,
> а показывает экран с инструкцией, чего именно не хватает.

## Активация аккаунтов и пароли
Сотрудник может задать или сбросить пароль сам, без участия владельца:
1. На экране входа — «Забыли пароль?», ввести email.
2. На почту придёт ссылка; по ней открывается экран «Задайте пароль».
3. После сохранения сотрудник сразу входит.

Тот же механизм работает для активации нового аккаунта: создаёшь пользователя
в Supabase, а он задаёт пароль по ссылке из письма. Чтобы письма уходили,
проверь Authentication → Email Templates и настройки SMTP, а в Authentication →
URL Configuration укажи адрес приложения (Site URL) — на него ведёт ссылка.

## Активация / деактивация сотрудников
- В шапке у админа есть кнопка **«Сотрудники»** — панель управления:
  переименование, выдача/снятие роли админа, включение/выключение доступа.
- **Деактивированный** сотрудник может войти, но не может создавать или менять
  брони — это гарантирует RLS на уровне базы (`current_is_active()`), не только UI.
- Свою роль и свою активность админ менять не может (защита от самоблокировки).

## Деплой на Vercel
1. Подключи репозиторий в Vercel. Если приложение лежит в подпапке
   (`cafe-booking/`), укажи её как **Root Directory**.
2. Framework определится как Vite автоматически (см. `vercel.json`):
   build `npm run build`, output `dist`, SPA-rewrites на `index.html`.
3. В Settings → Environment Variables добавь `VITE_SUPABASE_URL` и
   `VITE_SUPABASE_ANON_KEY`, затем задеплой.
4. Скопируй итоговый домен в Supabase → Authentication → URL Configuration
   (Site URL / Redirect URLs), чтобы ссылки из писем вели на прод.

## Обновление уже развёрнутой базы
Если база была создана до появления активации сотрудников — выполни один раз
`supabase/upgrade-activation.sql` в SQL Editor (он идемпотентный). На свежей
установке всё уже включено в `migration.sql`.

## Что важно знать
- **created_by** проставляет база автоматически — официант не может подделать,
  кто создал бронь. Имя автора видно на каждой брони.
- **Защита от двойной брони** — exclusion constraint на уровне индекса БД,
  гонок нет даже при одновременном бронировании одного стола.
- **Удаление мягкое** (soft delete) — бронь помечается, а не стирается.
  Восстановить может админ через снятие deleted_at.
- **Журнал действий** — таблица bookings_audit, читает только админ.
- Координаты и вместимость столов поправь в `supabase/migration.sql`
  (блок insert into tables) под реальный зал.
